01

Change Data Capture

Reading the database write-ahead log

Debezium

WAL-Based Change Data Capture

  • Reads the PostgreSQL WAL — no polling, no triggers
  • Every row change becomes a Kafka event
  • Initial snapshot captures existing data
  • Schema changes detected automatically from WAL
-- This INSERT...
INSERT INTO inventory.customers
  VALUES (1005, 'Alice', 'Smith',
         'alice@example.com');

-- ...becomes a Kafka event on topic
-- dbserver1.inventory.customers
-- with full Avro schema in Registry
02

Schema Governance

Apicurio Registry as the contract layer

Apicurio Registry

The Schema Contract Layer

Producer Side

Debezium auto-registers Avro schemas when it detects source schema changes

Compatibility Rules

BACKWARD compatibility prevents breaking changes from reaching consumers

Consumer Side

Iceberg sink deserializes using registry schemas and evolves table DDL automatically

CNCF Sandbox Avro JSON Schema Protobuf
03

The Lakehouse

Apache Iceberg on object storage

Apache Iceberg

Open Table Format

  • Parquet files on S3-compatible storage (MinIO)
  • ACID transactions — concurrent reads and writes
  • Schema evolution — add/rename/drop columns safely
  • Time travel — query any historical snapshot
  • Partition evolution — change partitioning without rewriting data
-- Query real-time data via Trino
SELECT * FROM iceberg.inventory.customers
ORDER BY id;

-- Time travel: query historical state
SELECT * FROM iceberg.inventory.customers
FOR VERSION AS OF 123456789;

Automatic Schema Evolution

-- 1. Add a column in PostgreSQL
ALTER TABLE inventory.customers
  ADD COLUMN phone VARCHAR(20);

-- 2. Debezium detects the WAL change
-- 3. Apicurio Registry registers new schema version
--    (BACKWARD compatible ✓)
-- 4. Iceberg sink evolves the table DDL automatically

-- 5. Query the new column in Trino
SELECT id, first_name, last_name, phone
FROM iceberg.inventory.customers
WHERE id = 1005;

--  id  | first_name | last_name | phone
-- -----+------------+-----------+--------------
-- 1005 | Alice      | Smith     | +34-555-0001

Demo

Iceberg Time Travel

-- List all snapshots
SELECT snapshot_id, committed_at
FROM iceberg.inventory."customers$snapshots"
ORDER BY committed_at;

-- Query data before Alice was added
SELECT * FROM iceberg.inventory.customers
FOR VERSION AS OF <snapshot_id>;

-- Only 4 original customers — Alice doesn't exist yet

Guidance

When Does This Replace Batch ETL?

Use CDC + IcebergKeep Batch ETL
Analysts need fresh data (minutes, not hours)Daily aggregation reports are sufficient
ML models need real-time featuresLow-volume, infrequently-changing data
Schema changes happen frequentlySchema is stable and rarely evolves
Multiple consumers need the same CDC streamSingle-use data extracts
You need full change history (audit, compliance)Only the latest state matters
